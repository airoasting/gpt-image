# gpt-image · 7원칙 이미지 프롬프트 설계

한국어로 떠올린 장면을 GPT 이미지 모델이 정확히 그릴 수 있는 프롬프트로 옮기는 스킬이다. 핵심은 자유로운 문장을 **7개 블록**으로 구조화해, 모델이 결과물의 종류·피사체·구도·맥락·스타일·빛·정확성 조건을 오해 없이 읽게 만드는 데 있다.

이 스킬의 기본 산출물은 이미지가 아니라 **바로 복사해서 쓰는 7블록 프롬프트 텍스트**다. 실제 생성은 선택 사항이다(마지막 섹션).

## 왜 7블록인가

이미지 모델의 실패는 대부분 "예쁘게", "고급스럽게", "감성적으로" 같은 모호한 형용사에서 나온다. 모델은 형용사를 상상으로 채우고, 사용자의 의도와 어긋난다. 7블록은 그 빈칸을 **결정 가능한 항목**으로 바꾼다. 결과물 종류를 첫 줄에 못 박고, 피사체 하나를 고정하고, 비율·시점을 숫자로 지정하고, 이미지 속 글자를 따옴표로 확정하면, 같은 프롬프트가 몇 번을 돌려도 의도에 수렴한다. 사용자가 원하는 건 "괜찮은 이미지 한 장"이 아니라 "이 의도가 반복 가능하게 나오는 프롬프트"라는 점을 기억한다.

## 작업 루프

1. **의도 파악과 결과물 유형 확정.** 사용자의 한 줄 요청을 아래 10개 카테고리 중 하나로 분류한다. 애매하면 가장 가까운 하나를 고르고, 정말 갈리면 한 가지만 짧게 되묻는다(예: "제품 광고컷인가요, 브랜드 무드컷인가요"). 이미 명확하면 묻지 말고 바로 설계한다.
2. **갤러리에서 유사 패턴 찾기.** 1차로 `docs/js/gallery-data.js`에서 같은 카테고리의 실제 사이트 예시를 찾는다(`"category": "<카테고리>"`로 검색하면 한 항목당 `title`·`image`·`prompt`가 함께 나온다). 이 예시들은 이미 7블록 한국어 형식으로 완성되어 있어 밀도와 어휘를 그대로 참고할 수 있다. 카테고리 특유의 세부 기법(다이어그램 문법, 이색 화풍, 편집 디자인 위계 등)이 더 필요하면 2차로 "10 결과물 카테고리" 표가 가리키는 `references/gallery-*.md`를 1개 더 읽는다. 처음부터 맨손으로 쓰지 않는다.
3. **7원칙으로 각 블록을 채운다.** 아래 "7가지 원칙"의 약한 예→강한 예 대비를 기준으로, 모든 블록을 구체 명사와 배치·수치로 채운다.
4. **필드 어휘로 정밀화.** 비율·스타일·조명·카메라·정확성 조건은 아래 "필드 어휘"의 정본 표현을 그대로 쓴다. 이 어휘는 사이트 프롬프트 빌더와 1:1로 맞춰져 있다.
5. **craft로 보정.** 이미지 속 글자, 다이어그램/인포그래픽, UI 스펙, 편집 포스터 위계, 멀티패널 일관성, 한글·중문 밀집 레이아웃이 걸리면 `references/prompt-craft.md`의 해당 절을 읽고 규칙을 적용한다.
6. **자가 점검 + 검증기.** 아래 "품질 체크리스트"를 통과할 때까지 다듬고, 고가치 산출물은 `python3 scripts/check_prompt.py <파일>`로 규칙 위반을 기계 검증한다(`ok: true` 확인).
7. **(선택) 실제 생성.** 사용자가 이미지 자체를 원하고 생성 수단이 있으면 마지막 섹션 절차로 실행한다. 아니면 프롬프트만 전달한다.

빠른 경로: 요청이 이미 정밀하고 사용자가 "바로 만들어"라고 하면, 갤러리·craft를 가볍게 확인하고 곧장 7블록으로 조립한다.

고비용·고완성도·모호한 요청(대형 포스터, 브랜드 키비주얼, 방향이 갈리는 콘셉트)에서는 조립 전에 1~2개 방향을 한 줄씩 제시하고 한 가지만 고르게 한다. 정밀한 요청에는 이 확인을 생략한다.

## 특수 요청: 프롬프트 개선과 레퍼런스 편집

description이 약속하는 두 갈래다. 백지 작성과 절차는 같되 시작점이 다르다.

**이미 쓴 프롬프트를 다듬어/고쳐 달라고 할 때.** 사용자의 프롬프트를 7블록에 대입해 어느 블록이 비었거나 모호한지 진단하고, 사용자가 이미 넣은 구체 정보(색·수치·고유명사)는 보존한 채 7블록으로 재구성한다. 무엇을 왜 바꿨는지 한두 줄로만 밝힌다. 원래 없던 정보는 지어내지 않는다.

**레퍼런스 이미지를 편집(변형/인페인트)할 때.** 먼저 유지할 불변 요소(인물·제품의 정체성, 구도, 조명 방향, 중요한 디테일)를 명시하고, 바꿀 부분만 정확히 기술한다. `references/prompt-craft.md`의 편집 불변식 절을 참고한다. 실제 생성은 호스트 이미지 도구의 레퍼런스·마스크(인페인트) 기능으로 수행한다(마지막 섹션).

## 핵심 산출물: 7블록 프롬프트 템플릿

아래 형식을 **정본**으로 삼는다. 블록 제목과 순서를 그대로 유지한다.

```
결과물 유형:
{카테고리 기준 한 줄. 예: 브랜드 출시용 3:4 세로형 제품 포스터}

주 피사체:
{화면 중심 대상 1개 + 보조 요소의 역할 + 유지할 형태·색상}

구도와 비율:
{비율}. {피사체 위치, 여백, 시점의 큰 틀}

맥락과 배경:
{사용 상황, 장소, 시간대, 분위기, 주변 요소}

스타일과 매체:
{사진 / 3D 렌더 / 수채화 / 포스터·편집 / 일러스트 / 픽셀 / 시네마틱 중 하나의 구체 표현}

빛과 디테일:
{조명 한 줄}
{카메라 시점 한 줄}

정확성 조건:
{완성도·텍스트·편집 유지 조건}
따옴표 안의 문구는 철자와 띄어쓰기를 그대로, 각 글자는 한 번씩만 또렷하게 렌더합니다. 전체 이미지는 여백과 위계가 정돈된, 하나의 완성된 장면처럼 자연스럽고 구체적인 마감으로 보여야 합니다.
```

마지막 두 문장(따옴표 텍스트 가드 + 긍정형 마감)은 카테고리와 무관하게 항상 붙인다. 이미지 속 글자가 없으면 따옴표 문장은 생략해도 된다. 빼고 싶은 요소는 "~ 없이/피하기"로 쓰지 말고 원하는 상태를 긍정형으로 재서술한다(아래 §네거티브 대신 긍정형).

## 네거티브 대신 긍정형 (중요)

GPT 이미지 모델은 장면 네거티브("군중 없이", "로고 없이")를 오히려 **렌더할 수 있다**. 빼고 싶은 것은 지우라고 말하지 말고 **원하는 상태를 긍정형으로 재서술**한다.

| 빼려는 것 | 나쁨 | 좋음(긍정형) |
|---|---|---|
| 여러 명 | "사람 여럿 말고" | "화면엔 주인공 한 사람만, 나머지 공간은 비운다" |
| 어수선한 배경 | "지저분하지 않게" | "한 가지 색조의 평평한 배경, 여백 위주" |
| 로고·워터마크 | "로고 없이" | "표면에 아무 표기 없이 매끈하게" |

**유일한 예외 = 텍스트 렌더 가드.** 이미지 안에 따옴표 카피가 실제로 있을 때만, 가독성 한 줄을 붙인다: "모든 글자는 한 번씩만 또렷하게." 그 밖의 `Avoid`, `No ~`, `Negative:` 라벨은 쓰지 않는다. 상세는 `references/prompt-craft.md` §14.

## 출력 규칙 (언어·밀도·전달)

- **언어.** 기본은 한국어 프롬프트다. 사이트 빌더와 일치하고, GPT 이미지 모델은 한국어를 잘 읽는다. 다만 (a) 사용자가 영어를 원하거나, (b) 갤러리의 특정 영어 스타일을 그대로 재현하거나, (c) 이미지 속 영어 텍스트가 많은 경우에는 영어 산문으로 낸다. **한 프롬프트 안에서 두 언어를 섞지 않는다.** 이미지 속 글자는 언어와 무관하게 원하는 표기 그대로 따옴표에 넣는다.
- **밀도.** 각 블록은 짧은 파편이 아니라 밀도 있는 한두 문장으로 쓴다. 아래 "완결 예시"의 사이트 갤러리 실제 프롬프트가 그렇다. 블록 제목은 뼈대일 뿐, 모델이 실제로 읽는 것은 그 아래 구체 묘사다.
- **전달.** 복사용 7블록을 펜스 코드블록으로 **먼저** 보여준다. 그다음 두 줄 이내로 무엇을 바꿔 끼울 수 있는지(비율·톤·스타일 스왑)만 덧붙인다. 카테고리 선택 이유는 한 줄이면 충분하다. 프롬프트를 길게 해설하지 않는다.

## 배치·양산 모드 (시안 여러 개)

"시안 여러 개", "차별화되게", "컨셉부터", "무드만 바꿔" 요청은 옮겨 적기가 아니라 발상이다. `references/variation-recipes.md`를 열어 **레시피 하나(사조 갈아끼우기·상반 요소 배치·무드→팔레트·감정→시각 신호·같은 소재 다른 구도)만 골라 반복**한다. 바꾸지 않을 블록은 고정하고 한 가지만 N개 값으로 벌린다. 한 번에 한 가지만 바꾼다(둘을 동시에 흔들면 세트감이 깨진다).

전달은 둘 중 하나:
- **엔트리 목록**: 시안마다 `제목 / 카테고리 / 축·값 / 7블록 프롬프트`.
- **jsonl(생성 연동 시)**: 한 줄 = 한 컷 = 한 생성. 스키마:
  ```json
  {"id":"cut-01","title":"...","category":"포스터","recipe":"사조:스위스","size":"1536x2048","prompt":"결과물 유형:\n..."}
  ```
  `size`는 필드 어휘의 출력 크기 표 값만 쓴다. 한 장에 한 컷씩 별도 행으로 낸다.

## 완결 예시 (사이트 갤러리 실제 항목)

아래 둘은 `docs/js/gallery-data.js`에 실제로 있는 사이트 갤러리 항목을 그대로 옮긴 것이다(지어낸 예시가 아니다). 형식·밀도·어휘를 그대로 기준으로 삼는다.

**예시 1: 제품·브랜드 (`ref-51`, 이미지 `docs/assets/generated-gallery/gallery-051.png`)**

> "전개도 기반 입체 제품 상자"

```
결과물 유형:
3:4 세로형 상업용 제품 이미지 또는 브랜드 비주얼. 주제는 "전개도 기반 입체 제품 상자"입니다. 완성 이미지는 제품 판매 또는 브랜드 제안서에 바로 넣을 수 있는 품질이어야 하며, 형태와 재질을 오해 없이 보여야 합니다.

주 피사체:
평면 전개도에서 완성된 입체 제품 상자를 보여주는 패키지 목업. 왼쪽에는 펼쳐진 전개도, 오른쪽에는 접힌 완성 상자와 작은 라벨 영역을 배치합니다. 중심 피사체의 형태, 위치, 행동이 먼저 읽히고 보조 요소는 주제를 설명하는 단서로만 사용합니다.

구도와 비율:
3:4 세로형 상업용 제품 이미지 또는 브랜드 비주얼. 제품이 가장 먼저 보이도록 중앙 축과 받침 그림자를 정리하고, 보조 소품은 재질과 사용 상황을 설명하는 정도로 제한합니다. 라벨 영역은 정면에서 읽히게 둡니다.

맥락과 배경:
깨끗한 흰 배경, 정확한 모서리, 무광 종이 질감, 인쇄 디자인이 실제 패키지처럼 보이게 합니다. 배경은 주 피사체를 설명하는 근거가 되어야 하며, 불필요한 장식으로 시선을 빼앗지 않습니다.

스타일과 매체:
상업용 제품 사진 또는 제품 렌더 수준의 마감. 정확한 재질, 반사, 모서리, 포장 구조, 브랜드 무드를 광고 이미지처럼 정리합니다.

빛과 디테일:
```

## 7가지 원칙 (정본)

각 원칙은 "약한 예 → 강한 예"로 익힌다. 아래는 하나의 예(노란 텀블러)를 7단계로 정밀화한 것이다. 실제 작업에서는 사용자의 소재로 같은 변환을 수행한다.

1. **결과물 유형을 정하기.** 포스터·인물 사진·UI·인포그래픽처럼 종류를 첫 줄에 고정한다.
   - 약함: "노란 텀블러가 있는 멋진 카페 느낌"
   - 강함: "선샤인 옐로우 세라믹 텀블러의 브랜드 출시용 3:4 세로형 제품 포스터. 제품이 중앙에 크게 보이는 인쇄물 스타일"
2. **주 피사체를 고정하기.** 중심 대상·인물·제품·장면을 하나로 분명히 한다.
   - 약함: "봄 카페 분위기, 꽃, 디저트, 음료, 따뜻한 햇살"
   - 강함: "선샤인 옐로우 세라믹 텀블러를 화면 중앙에 크게 배치. 뒤에는 봄꽃과 카페 배경을 부드럽게 흐리게 처리"
3. **구도와 비율을 지정하기.** 화면 비율, 피사체 위치, 여백, 시점의 큰 틀을 정한다.
   - 약함: "노란 텀블러를 들고 여행 간 느낌. 적당히 예쁜 사진"
   - 강함: "16:9 가로형 와이드 제품 여행 사진. 노란 텀블러는 오른쪽 1/3 지점, 왼쪽에는 호수와 산맥의 넓은 여백"
4. **맥락과 배경을 설명하기.** 어디에서, 어떤 상황에서, 어떤 분위기로 보일지 정한다.
   - 약함: "노란 텀블러를 좋은 분위기에서 보여주기"
   - 강함: "일요일 아침 주방 창가에서 쓰는 노란 텀블러. 반쯤 열린 노트, 커피 원두, 부드러운 아침 햇살로 차분한 루틴을 표현"
5. **스타일과 매체를 선택하기.** 사진·3D 렌더·수채화·포스터·편집 디자인처럼 표현 방식을 정한다.
   - 약함: "노란 텀블러를 고급스럽게. 예쁘고 세련된 느낌"
   - 강함: "무광 세라믹 질감이 보이는 노란 텀블러의 고급 제품 렌더. 부드러운 스튜디오 조명, 석회석 받침대, 인디고 포인트"
6. **빛과 디테일을 구체화하기.** 조명·질감·재료·표면·카메라 렌즈처럼 완성도를 좌우하는 요소를 쓴다.
   - 약함: "노란 텀블러를 극적으로 찍기. 분위기 있게"
   - 강함: "노란 텀블러를 바닥 가까운 낮은 앵글로 촬영. 강한 림 라이트와 젖은 돌 바닥의 노란빛 반사를 사용"
7. **정확성 조건을 덧붙이기.** 텍스트 가드, 유지 요소, 원하는 마감을 긍정형으로 마지막에 쓴다. 빼고 싶은 것은 "없이/피하기" 대신 원하는 상태로 재서술한다.
   - 약함: "노란 텀블러 포스터를 깔끔하게. 제목도 적당히 넣기"
   - 강함: '노란 텀블러 포스터. 제목 "Morning Solar"는 각 글자 한 번씩 또렷하게. 뚜껑은 닫힌 원래 형태 유지, 표면엔 표기 없이 매끈하게.'

원칙을 관통하는 규칙: **모든 형용사는 명사·수치·배치로 환원한다.** "고급스럽게"는 "무광 세라믹 + 석회석 받침대 + 스튜디오 조명"으로, "극적으로"는 "낮은 앵글 + 림 라이트"로 바꾼다. `8k·uhd·masterpiece·best quality` 같은 낡은 품질 토큰과 "어워드 수준으로·전문가처럼" 같은 막연한 기준도 넣지 않는다. 기준이 프롬프트 밖에 있으면 평균값만 나온다.

## 필드 어휘 (정본)

블록 3·5·6·7을 채울 때 아래 표현을 그대로 쓴다. 사용자가 다른 값을 원하면 존중하되, 기본값은 이 어휘를 따른다.

**구도와 비율 (블록 3)**
- `3:4 세로형` · `16:9 가로형` · `1:1 정사각형` · `9:16 모바일 세로형` · `21:9 와이드`
- **출력 크기 표(생성용 픽셀)**: 1:1 → `1024x1024`, 3:4 → `1536x2048`, 16:9 → `1920x1080`, 9:16 → `1080x1920`, 21:9 → `2520x1080`. 사이트 메타데이터와 같은 값이다. 글자가 많거나 컷이 밀집하면 긴 변을 크게(예: `2048`) 잡아 작은 글자의 판독을 확보한다.

**색 팔레트 (블록 4·5 보강)**
- 색감은 형용사 대신 **HEX 3~5개**로 박는다(예: `#1B2566 딥 인디고 / #F2B84B 골드 / #FAF1E2 페이퍼`). 60/30/10 비율감을 함께 쓰면 위계가 선다.

**스타일과 매체 (블록 5)**: 카테고리 기본값을 덮어쓸 때
- 실사 사진: 실제 카메라 질감, 자연스러운 그림자, 현실적 배경 깊이
- 3D 제품 렌더: 표면 재질·반사·형태 정확성, 광고 이미지 마감
- 포스터/편집: 명확한 시각 위계, 넉넉한 여백, 인쇄물 완성도
- 일러스트레이션: 선·색면·질감·장식 요소가 주제와 조화
- 수채화: 종이 질감, 투명한 색감, 자연스러운 번짐, 부드러운 붓터치
- 픽셀 아트: 제한된 팔레트, 또렷한 실루엣, 작은 해상도에서도 읽히는 형태
- 시네마틱: 영화 한 프레임처럼 빛·색·공간감·존재감

**조명 (블록 6, 첫 줄)**

- 최종 결과물: 글자 가독성·구도·색감·디테일 일관성 우선
- 아이디어 탐색: 완벽한 마감보다 다양한 시각 가능성·분위기 실험 허용
- 텍스트/라벨 중심: 따옴표 안 문구를 정확·선명·충분히 크게, 깨진 글자 금지
- 레퍼런스 편집: 바꾸라 한 부분 외 정체성·구도·조명 방향·중요 디테일 유지

## 검증기 (규칙 자동 점검)

전달·생성 전에 프롬프트를 코드로 검증한다. 주관적 체크리스트를 실행 가능한 게이트로 바꾼다.

```bash
python3 scripts/check_prompt.py <프롬프트파일>      # 또는 stdin 파이프
```

검출: 7블록 누락·순서, 모호 형용사(예쁘게·고급스럽게·감성적으로·멋지게·세련되게), 화면비 토큰 누락 또는 5종 밖, 낡은 품질 토큰(8k·uhd·masterpiece), 장면 네거티브(Avoid/No ~/Negative:), 한 프롬프트 내 언어 혼용, 따옴표 텍스트에 가독성 가드 누락. 출력은 `{ok, errors[], warnings[]}` JSON이고 errors가 0이어야 통과다.

규칙 근거는 실측이다: 카테고리당 1개(10컷) 비전 채점 결과 프롬프트-이미지 정합 goal_fit 5.0/5, 한글 텍스트 4.5/5, 꼬리 네거티브 렌더 0/10(→ 한국어 배제형은 warning). 상세 `experiments/REPORT.md`, 채점 기준 `experiments/RUBRIC.md`.

## 레퍼런스 로딩 정책

**1차: 사이트 갤러리.** `docs/js/gallery-data.js`에서 `"category": "<카테고리>"`를 검색해 같은 카테고리의 실제 예시 2~3개를 확인한다. 각 항목은 `title`(주제)·`image`(실제 이미지 경로)·`prompt`(완성된 7블록 한국어 프롬프트)를 담고 있다. 형식·밀도·어휘를 그대로 참고할 최우선 소스다. 대부분의 요청은 이 단계만으로 충분하다.

**2차: 레퍼런스 파일.** 한 카테고리의 예시를 이미지 미리보기와 함께 죽 훑어보거나, 세부 기법(다이어그램 문법, 이색 화풍, 편집 위계, 니치 소재)이 필요할 때 연다.
- `references/gallery.md`: 10개 카테고리 라우팅 인덱스.
- `references/gallery-*.md`: 10개 카테고리 파일. 각 파일에 그 카테고리의 완성 프롬프트 10개(사이트 갤러리와 동일한 한국어 프롬프트)와 이미지 미리보기가 들어 있다. 1개(혼합이면 2~3개)만 읽는다.
- `references/variation-recipes.md`: 변주 레시피(사조 갈아끼우기·상반 요소 배치·무드→팔레트·감정→시각 신호·같은 소재 다른 구도). "차별화되게·컨셉부터·시안 여러 개·무드만 바꿔" 요청일 때만 연다.
- `references/prompt-craft.md`: 프롬프트 정밀화 체크리스트(이미지 속 글자, JSON형 프롬프트, 고정 영역 인포그래픽, 다이어그램 문법, UI 스펙, 멀티패널 일관성, 편집 불변식, 한글·중문 밀집 레이아웃). 해당 이슈가 있을 때만 해당 절을 연다.
- `references/openai-cookbook.md`: 공식 모델·파라미터 의미. 모델 역량이나 API 동작 질문일 때만 연다.

가장 작은 유용한 조각만 로딩한다. 사이트 갤러리(`gallery-data.js`)가 정본이고 레퍼런스 파일은 그 사본이니, 두 소스가 어긋나면 사이트 갤러리를 따른다.

## 품질 체크리스트 (9.5 게이트)

전달 전에 스스로 묻는다. 하나라도 아니오면 다시 다듬는다.

- 7개 블록이 모두 채워졌고 순서가 정본과 같은가.
- 첫 줄에서 결과물의 종류가 즉시 읽히는가.
- 주 피사체가 하나로 고정되어 있는가(둘 이상이 경쟁하지 않는가).
- "예쁘게·고급스럽게·감성적으로" 같은 모호한 형용사를 명사·수치·배치로 환원했는가.
- 비율이 숫자로 지정되었는가.
- 이미지 속 글자가 있으면 따옴표 안에 정확히 넣고 유지 조건을 붙였는가.
- 카테고리에 맞는 사이트 갤러리 예시를 최소 1개 참고했는가.
- 사용자 소재의 고유명사·색·형태를 지어내지 않고 요청에 근거했는가.
- 출력 언어가 하나로 통일되었고, 복사용 블록을 먼저 제시했는가.
- 빼려는 요소를 "없이/피하기"가 아니라 긍정형으로 재서술했는가(텍스트 가드 1줄만 예외).
- 낡은 품질 토큰(8k·masterpiece)과 막연한 기준("어워드 수준으로")이 없는가.
- `scripts/check_prompt.py`가 `ok: true`인가(고가치 산출물).

## (선택) 실제 이미지 생성

사용자가 프롬프트가 아니라 **이미지 자체**를 원할 때만 실행한다. 이 스킬은 프롬프트 설계가 핵심이며, 이미지 생성기를 번들하지 않는다.

1. **호스트 우선.** 실행 환경에 네이티브 이미지 생성 도구가 있으면 그것을 쓴다(별도 키 불필요). 최종 자산·이미지 속 글자·포스터·다이어그램·UI는 고화질로, 탐색·다량 변형은 저화질로 뽑는다.
2. **호스트에 없으면** 완성한 7블록 프롬프트를 ChatGPT의 이미지 생성 등 사용자가 쓰는 이미지 도구에 그대로 붙여넣게 안내한다. 결과가 어긋나면 후처리로 글자를 덧그리지 말고 프롬프트를 고쳐 재생성한다(`hooks/text-overlay-guard.sh`).

**안전 규칙**
- 이미지 생성 API 직접 호출은 사용량에 따라 과금될 수 있음을 사용자에게 먼저 알린다.
- 비밀값(API 키 등)을 파일에 쓰거나 화면에 출력하지 않는다.
- 사용자가 명시적으로 요청하지 않는 한 설치·`.env` 생성/수정·스킬 폴더 덮어쓰기를 하지 않는다.

## 하지 말 것

- 모호한 형용사를 그대로 프롬프트에 남기기.
- 7블록 순서를 임의로 바꾸거나 블록을 합치기.
- 사이트 갤러리·craft를 건너뛰고 맨손으로 쓰기(정밀 "바로 만들어" 요청은 예외).
- 요청에 없는 브랜드명·수치·문구를 지어내기.
- 요청받지 않은 이미지 생성 코드나 SDK 래퍼를 새로 작성하기.
- 생성된 이미지 위에 코드로 글자를 얹는 후처리(PIL/Pillow·ImageMagick·SVG/HTML 오버레이). 글자는 프롬프트로 이미지 안에서 렌더하고, 틀리면 프롬프트를 고쳐 재생성한다(캔버스 크기 ↑ → 카피 수 축소 순). 이 규칙은 `hooks/text-overlay-guard.sh`로도 강제된다.
